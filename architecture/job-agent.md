# Architecture: Job Application Agent

> Generated by /architecture on 2026-04-25
> Mode: standard (quick + deep dives on data, code structure, API)

## Architecture Pattern

**Modular Monolith.** Single process, clear module boundaries. The coordinator pattern enables future agents (apartments, recipes, games) as plugin folders under `agents/`. No microservices вҖ” this is a local desktop app for one user.

## Tech Stack

| Layer | Choice | Alternative | Reasoning |
|-------|--------|-------------|-----------|
| Desktop shell | Tauri 2.0 | Electron | Lightweight, secure, Rust-based. Decided in requirements. |
| Frontend | React + TypeScript | вҖ” | Tauri default. Decided in requirements. |
| Backend | Python + FastAPI | Flask | Async, auto-docs, type hints. Runs as Tauri sidecar process. |
| Database | SQLite (aiosqlite) | вҖ” | Local, zero-config, WAL mode for crash safety. |
| LLM | Multi-provider via abstraction | Claude (default), OpenAI, Ollama, HuggingFace, none | Users choose their provider. App works without any LLM (degraded mode). |
| Semantic similarity | Sentence Transformers (all-MiniLM-L6-v2) | вҖ” | Sentence/paragraph-level embeddings (~80MB). Handles "distributed systems" вүҲ "systems at scale". |
| Text similarity | scikit-learn (TF-IDF + cosine) | вҖ” | Keyword importance scoring. Lightweight, no model needed. |
| Entity extraction | spaCy (en_core_web_md) | NLTK | Pull companies, titles, skills from text (~50MB model). |
| Fuzzy matching | RapidFuzz | fuzzywuzzy | "React.js" вүҲ "ReactJS". Fast, no model needed. |
| HTML scraping | BeautifulSoup + trafilatura | вҖ” | BeautifulSoup for structured job board pages, trafilatura for general content extraction. |
| PDF export | WeasyPrint | reportlab | HTML/CSS вҶ’ PDF. Same templates for screen + print. |
| DOCX export | python-docx | вҖ” | Standard library for Word format. |
| PDF/DOCX import | PyMuPDF (fitz) | pdfplumber | Fast text extraction, handles complex layouts. |
| URL fetching | httpx | requests | Async HTTP client for fetching job posting pages. |
| HTML extraction | trafilatura | BeautifulSoup | Extracts main content from web pages (strips nav, ads, footers). |
| API key storage | keyring | env var | Cross-platform OS keychain access. No plaintext. |

## Decision Log

| # | Decision | Choice | Depends On | Rationale |
|---|----------|--------|-----------|-----------|
| 1 | Knowledge bank schema | Hybrid (typed tables + JSON metadata) | вҖ” | Balance of queryability and flexibility. Core entities get columns, extras go in JSON. |
| 2 | Generated docs storage | Content in DB, exports on filesystem | #1 | Markdown/HTML queryable in SQLite, binary exports (PDF/DOCX) stay as files. |
| 3 | Schema migrations | App-level auto-migrate | #1, #2 | Desktop app вҖ” users never run CLI. App checks PRAGMA user_version on startup, applies pending migrations silently. |
| 4 | Module organization | By Agent + Shared | вҖ” | `agents/job/` self-contained, `shared/` for DB, Claude, exporters. New agents = new folders. Matches coordinator pattern. |
| 5 | Design patterns | Strategy + Repository + Coordinator | #4 | Strategy for exporters, Repository for data access, Coordinator for agent routing. Each solves a real problem. |
| 6 | Prompt management | Python prompt modules | #4, #5 | Dedicated files in `agents/job/prompts/`. Testable, type-safe, keeps service code clean. |
| 7 | API style | REST | вҖ” | One client, clean resource boundaries, FastAPI native support. GraphQL unnecessary. |
| 8 | Endpoint design | Resource-based REST (see API Design section) | #7 | 6 resource groups matching feature groups. |
| 9 | Error handling | Structured error responses | #7 | `{"error": {"code": "...", "message": "...", "details": {...}}}`. Frontend can differentiate error types. |
| 10 | Streaming | No streaming (return complete) | #7 | Simpler to build. Quality over speed. Add streaming later if needed. |
| 11 | Job matching | Dedicated JobMatcher service (separate from JobParser) | #4, #1 | Parsing and matching are distinct responsibilities (SRP). Matcher uses existing repos (knowledge_repo, job_repo) вҖ” no new repo needed. Supports future proactive job search. |
| 12 | LLM strategy | Multi-provider with graceful degradation | #4, #5 | LLM abstraction layer (Strategy pattern) supports Claude, OpenAI, Ollama, HuggingFace, or none. Every AI feature has an algorithmic fallback. App is fully usable without any LLM вҖ” just less intelligent. |
| 13 | Job matching tech | Hybrid: algorithmic always + LLM enhancement when available | #11, #12 | Sentence Transformers for semantic similarity + TF-IDF + RapidFuzz runs always. LLM adds deep analysis for top matches when available. |
| 14 | Offline mode | Opt-in with system requirements warning + automated setup | #12, #13 | User selects offline mode вҶ’ warned about ~500MB disk, ~2GB RAM вҶ’ models downloaded automatically. |
| 15 | Match calibration | Judgement system: users rate match quality to tune scoring weights | #11, #13 | Stores anonymized calibration data locally. User judgements adjust matching weights over time. Exportable without PII. |

## Data Architecture

### Schema (Hybrid: typed tables + JSON metadata)

```sql
-- Versioning
-- PRAGMA user_version tracks current schema version

-- Knowledge Bank
CREATE TABLE experiences (
    id INTEGER PRIMARY KEY,
    type TEXT NOT NULL,              -- 'job', 'internship', 'contract', 'volunteer'
    title TEXT NOT NULL,
    company TEXT,
    start_date TEXT,                 -- ISO 8601
    end_date TEXT,                   -- NULL = current
    description TEXT,
    metadata JSON,                   -- flexible fields: location, remote, industry, etc.
    created_at TEXT DEFAULT (datetime('now')),
    updated_at TEXT DEFAULT (datetime('now'))
);

CREATE TABLE skills (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL UNIQUE,
    category TEXT NOT NULL,          -- 'language', 'framework', 'tool', 'soft_skill', 'domain'
    proficiency TEXT,                -- 'beginner', 'intermediate', 'advanced', 'expert'
    source_experience_id INTEGER REFERENCES experiences(id),
    created_at TEXT DEFAULT (datetime('now'))
);

CREATE TABLE achievements (
    id INTEGER PRIMARY KEY,
    experience_id INTEGER REFERENCES experiences(id),
    description TEXT NOT NULL,
    metric TEXT,                     -- "reduced latency by 40%"
    impact TEXT,                     -- "saved $50K/year"
    metadata JSON,
    created_at TEXT DEFAULT (datetime('now'))
);

CREATE TABLE education (
    id INTEGER PRIMARY KEY,
    institution TEXT NOT NULL,
    degree TEXT,
    field TEXT,
    start_date TEXT,
    end_date TEXT,
    metadata JSON,                   -- GPA, honors, relevant coursework
    created_at TEXT DEFAULT (datetime('now'))
);

CREATE TABLE projects (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,
    description TEXT,
    tech_stack JSON,                 -- ["Python", "React", "PostgreSQL"]
    url TEXT,
    metadata JSON,                   -- role, team size, status
    created_at TEXT DEFAULT (datetime('now'))
);

-- Jobs
CREATE TABLE jobs (
    id INTEGER PRIMARY KEY,
    title TEXT NOT NULL,
    company TEXT,
    source_url TEXT,                 -- original job posting URL
    source_text TEXT,                -- raw pasted text or fetched HTML content
    parsed_data JSON NOT NULL,       -- Claude-extracted: required_skills, preferred_skills,
                                     -- salary_range, location, remote_status, etc.
    match_score REAL,                -- 0.0-1.0 match against knowledge bank
    match_breakdown JSON,            -- {skills_match: 0.8, experience_match: 0.7, gaps: [...]}
    status TEXT DEFAULT 'saved',     -- 'saved', 'generating', 'applied', 'archived'
    created_at TEXT DEFAULT (datetime('now'))
);

-- Generated Documents
CREATE TABLE resumes (
    id INTEGER PRIMARY KEY,
    job_id INTEGER REFERENCES jobs(id),
    content TEXT NOT NULL,           -- markdown/HTML source
    preferences JSON NOT NULL,       -- {length: "1_page", tone: "formal", sections: [...], ...}
    export_path TEXT,                -- filesystem path to exported file
    export_format TEXT,              -- 'pdf', 'docx', 'txt', 'md'
    feedback INTEGER,                -- -1 (thumbs down), 0 (none), 1 (thumbs up)
    created_at TEXT DEFAULT (datetime('now'))
);

CREATE TABLE cover_letters (
    id INTEGER PRIMARY KEY,
    job_id INTEGER REFERENCES jobs(id),
    content TEXT NOT NULL,           -- markdown/HTML source (editable)
    preferences JSON,
    export_path TEXT,
    export_format TEXT,
    feedback INTEGER,
    created_at TEXT DEFAULT (datetime('now')),
    updated_at TEXT DEFAULT (datetime('now'))
);

-- Application Tracking
CREATE TABLE applications (
    id INTEGER PRIMARY KEY,
    job_id INTEGER REFERENCES jobs(id) NOT NULL,
    resume_id INTEGER REFERENCES resumes(id),
    cover_letter_id INTEGER REFERENCES cover_letters(id),
    status TEXT NOT NULL DEFAULT 'applied',  -- 'applied', 'interview', 'offer', 'rejected'
    notes TEXT,
    created_at TEXT DEFAULT (datetime('now')),
    updated_at TEXT DEFAULT (datetime('now'))
);

CREATE TABLE application_status_history (
    id INTEGER PRIMARY KEY,
    application_id INTEGER REFERENCES applications(id) NOT NULL,
    status TEXT NOT NULL,
    changed_at TEXT DEFAULT (datetime('now'))
);

-- User Preferences
CREATE TABLE preferences (
    id INTEGER PRIMARY KEY CHECK (id = 1),  -- singleton row
    defaults JSON NOT NULL                   -- {tone: "formal", length: "1_page", sections: [...], ...}
);

-- Match Calibration (Judgement System)
CREATE TABLE calibration_judgements (
    id INTEGER PRIMARY KEY,
    job_id INTEGER REFERENCES jobs(id),
    match_score REAL NOT NULL,           -- algorithmic score at time of judgement
    match_features JSON NOT NULL,        -- {skills_overlap: 0.7, semantic_sim: 0.6, tfidf: 0.5, ...}
    user_rating TEXT NOT NULL,           -- 'good', 'partial', 'poor'
    notes TEXT,                          -- optional user notes on why
    created_at TEXT DEFAULT (datetime('now'))
);

CREATE TABLE calibration_weights (
    id INTEGER PRIMARY KEY CHECK (id = 1),  -- singleton row
    weights JSON NOT NULL DEFAULT '{"skills_overlap": 0.3, "semantic_sim": 0.3, "tfidf": 0.2, "experience_years": 0.2}',
    updated_at TEXT DEFAULT (datetime('now'))
);

-- LLM Provider Configuration
CREATE TABLE llm_config (
    id INTEGER PRIMARY KEY CHECK (id = 1),  -- singleton row
    provider TEXT,                           -- 'claude', 'openai', 'ollama', 'huggingface', NULL (none)
    model TEXT,                              -- e.g., 'claude-sonnet-4-20250514', 'gpt-4o', 'llama3'
    base_url TEXT,                           -- for Ollama: 'http://localhost:11434', HF: endpoint URL
    config JSON                              -- provider-specific settings (temperature, max_tokens, etc.)
    -- API keys stored in OS keychain via keyring, NOT in DB
);
```

### Design Patterns

1. **Strategy Pattern** вҖ” Used in two places:
   - **Exporters** (`shared/export/`). Each format implements the `Exporter` protocol.
   - **LLM Providers** (`shared/llm/`). Each provider implements `LLMProvider` protocol. Services call `provider.complete(prompt)` without knowing if it's Claude, OpenAI, or Ollama.

2. **Repository Pattern** вҖ” Data access (`agents/job/repositories/`). Services call `repo.get_experiences()`, never write SQL directly.

3. **Coordinator Pattern** вҖ” Agent routing (`coordinator.py`). Routes `(agent_name, action, data)` to the right handler.

4. **Graceful Degradation Pattern** вҖ” Every service has two code paths:
   - **With LLM:** AI-powered (extraction, generation, semantic matching)
   - **Without LLM:** Algorithmic fallback (manual entry, templates, keyword matching)
   - Services check `if llm_provider:` and branch accordingly.

### Dependency Direction

```
Routes вҶ’ Services вҶ’ Repositories вҶ’ SQLite
                  вҶ’ LLMProvider (optional) вҶ’ Claude / OpenAI / Ollama / HuggingFace
                  вҶ’ Algorithms (always available) вҶ’ scikit-learn / spaCy
                  вҶ’ Exporters вҶ’ WeasyPrint / python-docx
                  вҶ’ Scraping вҶ’ httpx / BeautifulSoup / trafilatura

JobMatcher uses: knowledge_repo + job_repo + algorithms (always) + LLMProvider (optional)
JobParser uses:  job_repo + scraping (always) + LLMProvider (optional)
```

No circular dependencies. LLMProvider is always optional вҖ” services never crash if it's None.

### Degradation Matrix

| Feature | With LLM | Without LLM (offline) |
|---------|----------|----------------------|
| Knowledge bank input | Free text вҶ’ AI extracts structured data | Manual structured forms + spaCy entity extraction assists |
| Job parsing | AI extracts role, skills, salary, etc. | BeautifulSoup + regex + spaCy entity extraction |
| Job matching | Sentence Transformers + TF-IDF + RapidFuzz + LLM deep analysis on top N | Sentence Transformers + TF-IDF + RapidFuzz (still strong вҖ” handles semantic similarity) |
| Match calibration | Same вҖ” judgement system works identically in both modes | Same |
| Resume generation | AI-tailored content from knowledge bank | Template-based: selects entries by match score, fills templates |
| Cover letter generation | AI-drafted with tone/style control | Template-based: fill-in-the-blanks |
| Quality difference | High вҖ” nuanced, contextual, tailored phrasing | Good matching, functional docs вҖ” correct info but generic phrasing |

## Data Flow

### Resume Generation (Main Flow)

```
User pastes job URL(s)
    вҶ’ POST /api/jobs/parse {urls: ["...", "..."]}
    вҶ’ job_parser.py detects URL vs text per input
    вҶ’ httpx fetches pages, trafilatura/BeautifulSoup extracts content
    вҶ’ IF LLM available:
        вҶ’ LLM parses structured job data (role, skills, salary, etc.)
    вҶ’ IF no LLM:
        вҶ’ Regex/pattern extraction for common job board formats
        вҶ’ User can manually correct/fill missing fields in UI
    вҶ’ Save to SQLite (jobs table)
    вҶ’ Return parsed jobs to frontend

Auto-trigger matching (or user clicks "Match")
    вҶ’ POST /api/jobs/match-batch {job_ids: [1, 2, 3]}
    вҶ’ job_matcher.py reads knowledge bank via knowledge_repo
    вҶ’ ALWAYS: Algorithmic scoring (TF-IDF, cosine similarity, skill overlap)
    вҶ’ IF LLM available + top N matches:
        вҶ’ LLM deep analysis (semantic matching, gap analysis, career fit)
    вҶ’ Save match_score + match_breakdown to jobs table
    вҶ’ Return scored jobs sorted by match to frontend

User selects a job, clicks "Generate Resume"
    вҶ’ POST /api/resumes/generate {job_id: 1, preferences: {...}}
    вҶ’ resume.py reads knowledge bank from SQLite (via repository)
    вҶ’ IF LLM available:
        вҶ’ build_resume_prompt(knowledge, job, preferences)
        вҶ’ LLM generates tailored resume (markdown)
    вҶ’ IF no LLM:
        вҶ’ Template-based: select relevant entries by skill overlap
        вҶ’ Assemble resume from templates + knowledge bank data
    вҶ’ Save content to SQLite (resumes table)
    вҶ’ Return resume content to frontend

User clicks "Export as PDF"
    вҶ’ GET /api/resumes/1/export?format=pdf
    вҶ’ PDFExporter converts markdown вҶ’ HTML вҶ’ PDF (WeasyPrint)
    вҶ’ Save PDF to ~/.house-helper/exports/resumes/
    вҶ’ Return file download
```

### Knowledge Bank Input

```
User types/pastes experience text
    вҶ’ POST /api/knowledge/extract {text: "I worked at..."}
    вҶ’ Claude API extracts: experiences, skills, achievements
    вҶ’ Return structured data for review (NOT saved yet)

User reviews, edits, confirms
    вҶ’ POST /api/knowledge/entries {entries: [...]}
    вҶ’ Save to SQLite (experiences, skills, achievements tables)
    вҶ’ Return confirmed entries
```

### Error Flow

```
Any request fails
    вҶ’ Service catches exception
    вҶ’ Maps to error code (PARSE_FAILED, CLAUDE_API_ERROR, etc.)
    вҶ’ Returns structured error response
    вҶ’ Frontend shows contextual error message based on code

Claude API specifically fails
    вҶ’ Retry once with backoff
    вҶ’ If still fails, return CLAUDE_API_ERROR with details
    вҶ’ Frontend shows "Generation failed вҖ” try again" with the specific reason
```

## Engineering Principles Check

| Principle | Status | Notes |
|-----------|--------|-------|
| **SRP** | :white_check_mark: | Each service file has one job. Each exporter handles one format. Each LLM provider handles one API. Each algorithm module handles one technique. |
| **OCP** | :white_check_mark: | New agents = new folder. New exporters = new class. New LLM providers = new class. No existing code changes. |
| **LSP** | :white_check_mark: | All exporters implement `Exporter` protocol. All LLM providers implement `LLMProvider` protocol. All agents implement `Agent` interface. |
| **ISP** | :white_check_mark: | Repository interfaces are per-entity, not one mega-interface. |
| **DIP** | :white_check_mark: | Services depend on LLMProvider/Repository/Exporter abstractions, not concrete SDKs/SQLite/WeasyPrint directly. |
| **DRY** | :white_check_mark: | Shared Claude client, shared export logic, shared DB. No duplication across agents. |
| **KISS** | :white_check_mark: | No streaming, no GraphQL, no Alembic, no auth, no encryption. Simplest path. |
| **YAGNI** | :white_check_mark: | No cloud, no multi-user. LLM abstraction IS needed (multi-provider + degradation). Only what requirements specify. |
