## Architecture Pattern

Sidecar process — Tauri shell manages a PyInstaller-compiled Python backend as a subprocess. Frontend is bundled as static files. All communication via HTTP on localhost.

## Decision Log

| # | Decision | Choice | Depends On | Rationale |
|---|----------|--------|-----------|-----------|
| 1 | Backend bundling | PyInstaller binary | — | Zero Python install. Industry standard. ~150MB. |
| 2 | Frontend bundling | Tauri bundles dist/ | — | No Node at runtime. Offline capable. |
| 3 | Communication | HTTP on localhost:8040 | #1 | No IPC needed. API already exists. |
| 4 | Startup flow | Spawn sidecar → poll /health → show webview | #1, #3 | Simple, reliable, user sees splash during wait |
| 5 | Data location | ~/.kaarsaaz/ | — | Same as current. No change. |
| 6 | AUTH_MODE | Always "local" in Tauri | — | Desktop = single user, no login |

## Data Flow

### App Launch
```
User double-clicks app
  → Tauri main.rs: spawn Python sidecar binary
  → Show splash screen ("Starting...")
  → Poll GET http://localhost:8040/health every 200ms
  → /health returns 200 → hide splash, show webview
  → Webview loads bundled index.html (React app)
  → React app calls /api/* → Python backend responds
```

### App Close
```
User closes window (or Cmd+Q)
  → Tauri on_window_event: CloseRequested
  → Kill sidecar process (SIGTERM)
  → Wait 2s for graceful shutdown
  → If still alive, SIGKILL
  → App exits
```

## New File Structure

```

  src/

  tauri.conf.json     # App metadata, sidecar path, window size
  Cargo.toml          # tauri, serde dependencies
  icons/              # App icons (generated by tauri icon command)

  build-backend.sh    # PyInstaller: python → binary
  build-app.sh        # Full: backend + frontend + tauri
```

## Build Pipeline

### Local Development
```bash
# Terminal 1: backend (unchanged)
cd backend && uvicorn main:app --port 8040 --reload

# Terminal 2: tauri dev (starts Vite + Tauri window)
cargo tauri dev
```

### Production Build
```bash
# Step 1: Build Python backend binary
cd backend
pyinstaller --onefile --name kaarsaaz-backend main.py
cp dist/kaarsaaz-backend ../src-tauri/binaries/

# Step 2: Build frontend
cd frontend && npm run build

# Step 3: Build Tauri app
cargo tauri build
# Output: src-tauri/target/release/bundle/
#   macOS: House Helper.dmg
#   Windows: House Helper.exe
#   Linux: kaarsaaz.AppImage
```

### CI (GitHub Actions)
```yaml
matrix:
  - os: macos-latest    → .dmg
  - os: ubuntu-latest   → .AppImage
  - os: windows-latest  → .exe
```

Each builds: PyInstaller → npm build → cargo tauri build → upload artifact.

## Implementation Phases

### Phase 1: Tauri Shell (dev mode)
- Install Rust + Tauri CLI
- Create src-tauri/ with basic window config
- `cargo tauri dev` loads http://localhost:5173
- Backend started manually (existing workflow)
- **Test:** Tauri window opens, all features work in webview

### Phase 2: PyInstaller Binary
- Create PyInstaller spec for backend/main.py
- Include all Python deps (FastAPI, python-docx, etc.)
- Binary starts on port 8040
- Test on macOS first
- **Test:** Binary runs standalone, /health returns 200

### Phase 3: Integrated Sidecar
- Tauri spawns PyInstaller binary on launch
- Health check polling before showing webview
- Splash screen during startup
- Kill sidecar on close
- **Test:** Full lifecycle — launch, use features, close cleanly

### Phase 4: Cross-Platform CI
- GitHub Actions matrix for macOS/Windows/Linux
- Code signing (macOS notarization)
- GitHub Releases with downloadable artifacts
- **Test:** CI builds all 3 platforms successfully

## Engineering Principles Check

| Principle | Status | Notes |
|-----------|--------|-------|
| SRP | ✅ | Tauri = window. Python = logic. Clear boundary. |
| OCP | ✅ | New features don't require Tauri changes. |
| KISS | ✅ | HTTP on localhost — simplest integration possible. |
| YAGNI | ✅ | No Tauri IPC, no Rust commands. Shell only. |
| DRY | ✅ | Same codebase for web + desktop. |
