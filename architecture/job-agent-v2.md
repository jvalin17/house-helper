## Architecture Pattern

**Same Modular Monolith** — v1 architecture stays. V2 adds an automation pipeline layer on top of existing services. No structural changes to coordinator, repos, or shared modules.

## What's New in v2 (delta from v1)

| Layer | v1 | v2 Addition |
|-------|----|----|
| Pipeline | None | Auto-apply orchestrator: search → match → generate → confirm → apply → track |
| Job boards | Manual paste | Plugin-based: API aggregator + scrapers |
| Token management | None | Budget manager wrapping LLM providers |
| Resume guardrails | None | Evidence-only validation + user addition logging |
| Scheduler | None | APScheduler for periodic job search |
| Frontend | 4 tabs (manual) | 4 tabs (pipeline-focused) |

## Decision Log

| # | Decision | Choice | Depends On | Rationale |
|---|----------|--------|-----------|-----------|
| 1 | Job board access | APIs + scraping, both as plugins | — | Aggregator API (JSearch) for breadth, scrapers (BeautifulSoup) for LinkedIn/Indeed directly. Plugin interface supports both. |
| 2 | Plugin architecture | `JobBoardPlugin` protocol in `shared/job_boards/` | #1 | Each board implements `search(filters) → list[JobResult]`. Add new boards without touching existing code. |
| 3 | Auto-apply flow | Queue-based with state machine | — | pending → generating → ready → reviewing → confirmed → applied. User confirms each. Max 5 per batch. |
| 4 | Apply method | Email (mailto:) + browser (webbrowser.open) + manual | #3 | Email: compose in default mail app. Browser: open URL + docs ready. Manual: show URL + export docs. |
| 5 | Token budget | Wrapper around LLMProvider, per-feature tracking | — | `BudgetManager.complete()` checks budget before calling LLM. Tracks usage in `token_usage` table. Priority queue for features. |
| 6 | Frontend tabs | 4 tabs: Job Search → Resume Builder/Superpowers → Dashboard/Tracker → Settings | — | User's explicit ordering. Pipeline flows left-to-right. |
| 7 | Scheduler | APScheduler (in-process) | — | Lightweight, no message broker. Triggers search at user-configured intervals while app is open. |
| 8 | Resume guardrails | Evidence validation layer | — | Check every claim against knowledge bank. Log user additions with source="user_override". |
| 9 | Evidence logging | `evidence_log` table tracks source of all knowledge bank entries | #8 | Every entry tagged: resume_import, manual, user_override, llm_generated. |

## Auto-Apply Queue

### State Machine

```
                    ┌─────────┐
                    │ pending │ ← user selects jobs
                    └────┬────┘
                         │ generate docs
                    ┌────▼────┐
              ┌─────│generating│
              │     └────┬────┘
          failed         │ docs ready
              │     ┌────▼────┐
              │     │  ready  │
              │     └────┬────┘
              │          │ user opens review
              │     ┌────▼─────┐
              │     │reviewing │
              │     └──┬────┬──┘
              │   skip │    │ approve
              │  ┌─────▼┐ ┌▼────────┐
              │  │skipped│ │confirmed│
              │  └──────┘ └────┬────┘
              │                │ email/browser/manual
              │           ┌────▼────┐
              └──────────►│ applied │ → auto-track
                          └─────────┘
```

### Batch Apply API Flow

```
POST /api/apply/batch {job_ids: [1, 2, 3]}
  → Creates queue entries (status: pending)
  → Returns queue_id

GET /api/apply/queue/{queue_id}
  → Returns status of each job in the batch

POST /api/apply/generate/{queue_entry_id}
  → Generates resume + cover letter (checks token budget)
  → Status: pending → generating → ready

POST /api/apply/confirm/{queue_entry_id}
  → User confirms after review
  → Status: reviewing → confirmed

POST /api/apply/execute/{queue_entry_id}
  → Opens email/browser
  → Status: confirmed → applied
  → Creates application record in tracker

POST /api/apply/skip/{queue_entry_id}
  → Status: reviewing → skipped
```

## Token Budget Manager

```python
# shared/token_budget.py

class BudgetManager:
    def __init__(self, llm_provider, conn):
        self._llm = llm_provider
        self._conn = conn

    async def complete(self, prompt, feature, system=None):
        """LLM call with budget enforcement."""
        # 1. Check budget
        remaining = self.get_remaining_today()
        estimated = self.estimate_tokens(prompt)
        if estimated > remaining:
            raise BudgetExceededError(remaining, estimated)

        # 2. Call LLM
        result = await self._llm.complete(prompt, system=system)

        # 3. Track usage
        self.log_usage(feature, tokens_used, estimated_cost)
        return result

    def get_remaining_today(self) -> int: ...
    def estimate_tokens(self, text) -> int: ...
    def log_usage(self, feature, tokens, cost): ...
    def get_usage_summary(self) -> dict: ...
    def check_budget(self, feature) -> bool: ...
```

## Engineering Principles Check

| Principle | Status | Notes |
|-----------|--------|-------|
| **SRP** | :white_check_mark: | Each new service has one job. BudgetManager wraps LLM. AutoSearch wraps job boards. AutoApply manages queue. |
| **OCP** | :white_check_mark: | Job board plugins add new sources without modifying existing code. Template system adds formats without changes. |
| **LSP** | :white_check_mark: | All job board plugins implement `JobBoardPlugin` protocol. All LLM providers implement `LLMProvider`. |
| **ISP** | :white_check_mark: | `JobBoardPlugin` has 4 focused methods. `BudgetManager` wraps without expanding LLM interface. |
| **DIP** | :white_check_mark: | Services depend on protocols (JobBoardPlugin, LLMProvider), not concrete implementations. |
| **DRY** | :white_check_mark: | Token tracking centralized in BudgetManager. Evidence logging centralized in evidence_repo. |
| **KISS** | :white_check_mark: | Apply uses mailto: + webbrowser.open — no Selenium complexity for MVP. |
| **YAGNI** | :white_check_mark: | Browser form-filling, background daemon, A/B testing all parked for post-MVP. |
