# Requirements: Search Filters, Budget, Dashboard Reset

> Generated by /requirements on 2026-04-28
> Mode: quick
> Scope: feature (5 related features for existing app)
> User level: 3 (developer)
> Project state: existing

## Problem Statement

The job search flow lacks configurable defaults, important filters (sponsorship, clearance), resume-based matching selection, real-time cost visibility, and a way to reset stale data.

## Core Requirement

Make the search-to-match-to-apply flow configurable and self-maintaining.

## Boundaries
- **Excluded:** Auto-apply execution (future), Playwright form filling
- **Constraints:** Must work with existing SQLite DB, max 5 saved resumes, existing profile table

## Functional Requirements

### Feature 1: Default Search Criteria

| Capability | Input | Output | Priority |
|-----------|-------|--------|----------|
| Save default role/title | Text input (e.g., "Senior Software Engineer") | Stored in profile/settings | must |
| Save default location | Text input (e.g., "United States") | Stored in profile/settings | must |
| Save default keywords | Comma-separated text | Stored in profile/settings | should |
| Pre-fill search filters from defaults | Page load | Filters populated | must |
| Auto-apply uses saved defaults | Pipeline trigger | Search runs with defaults | must |

**Where:** Settings section OR dedicated "Search Defaults" card in Job Search tab.
**Storage:** `profiles` table already has `search_title`, `search_keywords`, `search_location`, `search_remote` columns.

### Feature 2: Extra Job Filters (Profile-Level)

| Capability | Input | Output | Priority |
|-----------|-------|--------|----------|
| Sponsorship Required checkbox | Toggle on/off | Filter out jobs requiring sponsorship | must |
| Security Clearance checkbox | Toggle on/off | Filter out jobs requiring clearance | must |
| Internship Applicable checkbox | Toggle on/off | Filter out internship-only jobs | should |
| Filters saved per profile | Checkboxes | Persisted in profile | must |

**How filtering works:** Post-fetch scan of job description text for keywords:
- Sponsorship: "visa sponsorship", "sponsorship", "authorized to work", "US citizen"
- Clearance: "security clearance", "clearance required", "TS/SCI", "Secret clearance"
- Internship: "internship", "intern position", "co-op"

If checkbox is ON (user does NOT have sponsorship/clearance), filter OUT jobs containing those keywords.

**Where:** Profile section in Knowledge Bank OR a "Preferences" card in Job Search tab.

### Feature 3: Resume Selector for Matching

| Capability | Input | Output | Priority |
|-----------|-------|--------|----------|
| Dropdown showing saved resumes (max 5) | Select from list | Selected resume used for matching | must |
| Default resume pre-selected | Page load | Most recently saved resume selected | should |
| Match score uses selected resume's content | Resume selection | TF-IDF/semantic runs against selected resume text | must |
| LLM analysis uses selected resume | Resume selection | analyze_fit prompt receives selected resume | must |

**Where:** Dropdown in Job Search tab, near the search filters.

### Feature 4: Real-Time Cost Display

| Capability | Input | Output | Priority |
|-----------|-------|--------|----------|
| Show total cost incurred today | Auto-refresh | "$0.07 spent today" in Settings | must |
| Show per-feature breakdown | Auto-refresh | "Analysis: $0.04, Generation: $0.03" | should |
| Remove fixed $0.50 budget limit | Settings change | No limit enforcement, just display | must |
| Show all-time total cost | Settings page | "$1.23 total since start" | could |

**Data source:** `token_usage` table already tracks per-call cost. Just display it without enforcing a limit.

### Feature 5: Dashboard Reset

| Capability | Input | Output | Priority |
|-----------|-------|--------|----------|
| "Reset Dashboard" button | Confirmation dialog | Clears jobs, applications, generated resumes | must |
| Preserves knowledge bank | Reset action | Experiences, skills, education, projects untouched | must |
| Preserves settings | Reset action | LLM config, budget, API keys untouched | must |
| Preserves resume templates | Reset action | Uploaded DOCX templates untouched | must |
| Preserves saved resumes (is_saved=1) | Reset action | Only ephemeral resumes deleted | should |
| Confirmation dialog | Button click | "This will clear X jobs, Y applications. KB and settings preserved." | must |

**What gets deleted:** `jobs`, `applications`, `application_status_history`, `auto_apply_queue`, `resumes WHERE is_saved = 0`
**What stays:** `experiences`, `skills`, `education`, `projects`, `resume_templates`, `settings`, `resumes WHERE is_saved = 1`, `suggestion_feedback`, `calibration_judgements`

## Data Requirements

| Data | Source | Stored? | Table |

| Default location | User input | Yes | profiles.search_location |
| Sponsorship filter | Checkbox | Yes | profiles (new column or JSON in resume_preferences) |
| Clearance filter | Checkbox | Yes | profiles (same) |
| Selected resume for matching | Dropdown | Session only | Frontend state (not DB) |

## Assumptions

- Profile table already has `search_title`, `search_keywords`, `search_location`, `search_remote` columns
- Job description text is available in `jobs.parsed_data` for keyword filtering
- Saved resumes have `content` text that can be used for matching

## Implementation Order

1. **Feature 4 (Budget)** — simplest, just remove limit + display
2. **Feature 5 (Reset)** — one endpoint + confirmation dialog
3. **Feature 1 (Defaults)** — profile read/write + pre-fill
4. **Feature 2 (Filters)** — profile columns + post-fetch filtering
5. **Feature 3 (Resume selector)** — dropdown + wiring to match/analyze
