# Requirements: Tauri Desktop App

> Generated by /requirements on 2026-04-29
> Mode: quick
> Scope: infrastructure (packaging existing app as native desktop)
> User level: 3 (developer)
> Project state: existing

## Problem Statement

Users must install Python 3.12+ and Node 18+ manually, clone the repo, run setup.sh, and start two servers. This is too much for non-developers. The app should be a downloadable .dmg/.exe/.AppImage that just works.

## Core Requirement

Download one file, double-click, app runs. No Python, no Node, no terminal.

## Boundaries
- **Excluded:** Auto-update (future), app store distribution, mobile
- **Constraints:** Must work in AUTH_MODE=local (single user, no login). Backend bundled as standalone binary.

## Functional Requirements

### 1. Packaging

| Capability | Priority |
|-----------|----------|
| macOS .dmg with drag-to-Applications | must |
| Windows .exe installer (or portable .exe) | must |
| Linux .AppImage | must |
| Backend bundled as PyInstaller binary (no Python needed) | must |
| Frontend bundled in Tauri webview (no Node needed) | must |
| SQLite DB at ~/.kaarsaaz/ (same as now) | must |
| .env file for API keys (same as now) | must |
| App icon + name "Kaarsaaz" | should |
| First-launch setup wizard (enter API key) | could |

### 2. Backend Bundling

| Capability | Priority |
|-----------|----------|
| PyInstaller builds backend/main.py into standalone binary | must |
| All Python deps included (FastAPI, python-docx, etc.) | must |
| Binary starts uvicorn on localhost:8040 on app launch | must |
| Binary stops when app window closes | must |
| Ollama detection still works (checks localhost:11434) | must |
| Sentence Transformers included (optional — adds ~200MB) | could |

### 3. Frontend / Tauri Shell

| Capability | Priority |
|-----------|----------|
| Tauri 2.0 webview loads http://localhost:8040 (or bundled static) | must |
| Window title: "Kaarsaaz" | must |
| Native menu bar (File, Edit, Help) | should |
| Tray icon with "Open" / "Quit" | could |
| Window remembers size and position | could |

## Implementation Phases

### Phase 1: Tauri shell + dev mode (no PyInstaller)
- Set up Tauri 2.0 project in `src-tauri/`
- Tauri loads `http://localhost:5173` (Vite dev server)
- Backend still started manually
- Verify all features work in Tauri webview
- **Tests:** Tauri builds, webview renders, API calls work

### Phase 2: PyInstaller backend binary
- PyInstaller spec for `backend/main.py`
- Include all Python deps
- Binary starts on port 8040
- Test on macOS first, then cross-compile
- **Tests:** Binary starts, /health returns 200, API calls work

### Phase 3: Integrated launch
- Tauri starts PyInstaller binary as sidecar process
- Waits for backend to be ready (polls /health)
- Opens webview when ready
- Kills sidecar on window close
- **Tests:** Full lifecycle — launch, use, close

### Phase 4: Cross-platform builds
- GitHub Actions CI for macOS/Windows/Linux
- Code signing (macOS notarization, Windows signing)
- Release artifacts as GitHub releases
- **Tests:** CI builds for all 3 platforms

## Dependencies
- `tauri-cli` 2.0+ (Rust toolchain)
- `pyinstaller` 6.0+
- GitHub Actions runners for cross-platform builds

# Requirements: Browser Form Filling (Playwright Auto-Apply)

> Generated by /requirements on 2026-04-29
> Mode: quick
> Scope: feature (auto-apply pipeline completion)
> User level: 3 (developer)
> Project state: existing

## Problem Statement

The "auto-apply" pipeline currently opens job URLs in the browser but doesn't fill forms. Users must manually copy their resume info into each application. The goal is full automation — Playwright fills the form and submits.

## Core Requirement

For each job in the apply queue, Playwright opens the application page, fills all fields from the user's resume/KB, attaches the resume file, and submits. Fully automated.

## Boundaries
- **Excluded:** CAPTCHA solving (detect and pause for human), payment-gated applications
- **Constraints:** Must handle failures gracefully (don't crash on one bad form). Must log what was submitted. Must work with the existing auto_apply_queue table.

## Functional Requirements

### 1. Form Detection & Filling

| Capability | Priority |
|-----------|----------|
| Detect common form fields (name, email, phone, LinkedIn, resume upload) | must |
| Fill text fields from user's KB/profile | must |
| Upload resume PDF/DOCX to file input | must |
| Handle dropdowns (location, visa status, experience years) | must |
| Handle radio buttons (willing to relocate, etc.) | should |
| Handle multi-step forms (Next → Next → Submit) | must |
| Detect "Easy Apply" on LinkedIn | should |
| Handle cover letter paste/upload | should |

### 2. Job Site Support

| Site | Form Type | Priority |
|------|-----------|----------|
| LinkedIn Easy Apply | Modal flow, multi-step | must |
| Indeed Apply | Redirect to company site or Indeed form | must |
| Greenhouse | Standard ATS form (/jobs/123/apply) | must |
| Lever | Standard ATS form (/jobs/456/apply) | must |
| Generic company pages | Heuristic field detection | should |

### 3. Application Queue Integration

| Capability | Priority |
|-----------|----------|
| Read from auto_apply_queue (job_id, resume_id, cover_letter_id) | must |
| Export resume as PDF/DOCX to temp file for upload | must |
| Update queue status: pending → applying → applied/failed | must |
| Log each field filled + submission result | must |
| Screenshot after submission (proof of application) | should |
| Skip if CAPTCHA detected (mark as "needs_human") | must |
| Retry on network error (max 2 retries) | should |

### 4. Safety & Transparency

| Capability | Priority |
|-----------|----------|
| Dry-run mode: fill but don't submit (for testing) | must |
| Log every action (opened URL, filled field X with Y, clicked submit) | must |
| Never fake "applied" status — only mark when actually submitted | must |
| Store submission screenshot in user's data dir | should |
| Rate limit: max 1 application per 30 seconds | must |
| Daily limit: configurable max applications per day (default 20) | should |

## Implementation Phases

### Phase 1: Playwright infrastructure + generic form filler
- Install Playwright, set up browser context
- Build generic form field detector (label → input mapping)
- Fill name, email, phone from profile/KB
- Attach resume file to file inputs
- **Dry-run mode only** — fill but don't submit
- **Tests:** Field detection on sample HTML forms, correct values filled

### Phase 2: LinkedIn Easy Apply
- Handle LinkedIn login (user provides session cookie or logs in manually)
- Detect "Easy Apply" button, open modal
- Fill multi-step form (contact → resume → questions → submit)
- Handle LinkedIn-specific dropdowns and radio buttons
- **Tests:** Mock LinkedIn HTML, verify field mapping

### Phase 3: ATS systems (Greenhouse, Lever)
- Detect Greenhouse/Lever URLs (patterns: /jobs/*/apply)
- Map standard ATS fields to KB data
- Handle cover letter textarea
- Submit and verify confirmation page
- **Tests:** Mock ATS HTML forms, verify submission

### Phase 4: Generic company pages + Indeed
- Heuristic field detection (look for labels: "name", "email", "resume")
- Handle Indeed's redirect-to-company pattern
- Fallback: if can't detect form, open page and notify user
- **Tests:** Various form layouts, graceful failure on unrecognizable forms

### Phase 5: Integration with auto-apply pipeline
- Wire into ApplyPipeline.tsx "Deploy" stage
- Replace `sleep()` animation with real Playwright calls
- Update auto_apply_queue status in real-time
- Show live progress in the UI (which field is being filled)
- **Tests:** Full pipeline — search → match → generate → fill → submit

## Data Requirements

| Data | Source | Used For |
|------|--------|---------|
| Name | Profile/KB | Form filling |
| Email | Profile/KB | Form filling |
| Phone | Profile/KB | Form filling |
| LinkedIn URL | Profile/KB | Form filling |
| Resume PDF | Generated + exported | File upload |
| Cover letter | Generated | Text paste |
| Visa status | Profile preferences | Dropdown selection |
| Years of experience | KB (calculated) | Dropdown selection |
| Location | Profile | Dropdown/text field |

## Dependencies
- `playwright` Python package
- Chromium browser (auto-installed by Playwright)
- For LinkedIn: user session (cookie or manual login)

## Assumptions
- Playwright runs headless by default (no visible browser window)
- User has already generated a resume + cover letter for each job before applying
- The auto_apply_queue table already has resume_id and cover_letter_id populated
- LinkedIn may block automated submissions — need rate limiting and human-like delays
