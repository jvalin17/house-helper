## Problem Statement

Build an intelligent job application agent that automates the full pipeline: building a rich knowledge bank of work experience, generating tailored resumes and cover letters per job posting, and tracking application status — all running locally on desktop via Tauri.

## Core Requirement

**End-to-end automation with a knowledge-bank-first approach.** The agent's differentiator is that resumes are not static files — they are generated fresh each time from a deep, growing profile of the user's experience, tailored to each specific job posting and user preferences.

## Boundaries

- **Excluded:**
  - No payment features (no subscriptions, tiers, or billing)
  - No auto-submitting applications (prepare materials only, user submits manually)
  - No cloud storage — all data stays local (SQLite + local filesystem)
- **Constraints:**
  - Tech stack: Python (FastAPI) backend, React/TypeScript frontend, Tauri 2.0 desktop, Claude API, SQLite
  - Local-first architecture — no external servers for data storage
  - Quality over speed — longer generation time is acceptable for better output

## User Stories

- As a job seeker, I want to dump all my work experience freely so that the agent has a rich knowledge bank to draw from.
- As a job seeker, I want to paste a job posting and get a tailored resume generated from my knowledge bank so that each application highlights the most relevant experience.
- As a job seeker, I want to control resume format (length, sections, tone, layout, emphasis) so that each resume matches my preferences and the job's expectations.
- As a job seeker, I want a cover letter drafted and editable in the UI so that I can refine it before sending.
- As a job seeker, I want to track which applications led to interviews so that the agent learns which resume/cover letter approaches work best.

## Functional Requirements

### 1. Knowledge Bank

The foundation of the system. Users build a comprehensive profile that feeds all generation.

| Capability | Input | Output | Priority |
|-----------|-------|--------|----------|
| Free-text experience dump | Unstructured text about jobs, projects, achievements | Claude extracts structured data (titles, companies, dates, skills, accomplishments) | must |
| Structured review/edit | Extracted data presented in editable form | User-verified structured profile entries | must |
| Incremental enrichment | Additional text dumps over time | Knowledge bank grows richer, new entries merged with existing | must |
| Skills extraction | Work descriptions | Categorized skills list (languages, frameworks, tools, soft skills) | must |
| Quantified achievements | User descriptions like "improved performance" | Claude prompts for specifics: "by how much? what metric?" | should |
| Import existing resume | PDF/DOCX upload | Parsed and added to knowledge bank as starting point | should |
| Project portfolio | Descriptions of side projects, open source, etc. | Structured project entries with tech stack, impact, links | should |

### 2. Resume Generation

On-demand, tailored resume creation from the knowledge bank.

| Capability | Input | Output | Priority |
|-----------|-------|--------|----------|
| Job-tailored generation | Job posting + knowledge bank + preferences | Complete tailored resume | must |
| Length control | User preference (1 page, 2 page, etc.) | Resume constrained to specified length | must |
| Section control | Which sections to include/exclude (summary, skills, education, projects) | Resume with selected sections only | must |
| Tone control | Formal / conversational / technical | Resume matching specified tone | must |
| Emphasis control | Leadership vs IC vs technical depth | Content selection and framing adjusted | must |
| Layout/style preferences | Font style, ordering of sections, visual style | Formatted output matching preferences | should |
| Highlight/omit per job | User marks specific experience to include or exclude for this job | Customized content selection | should |
| Multiple format export | User selects format | PDF, DOCX, plain text, or Markdown output | must |
| Resume versioning | Each generated resume | Saved with metadata (which job, which preferences, date) | should |

### 3. Cover Letter Generation

AI-drafted, human-edited cover letters.

| Capability | Input | Output | Priority |
|-----------|-------|--------|----------|
| Auto-generate from job + profile | Job posting + knowledge bank + user preferences | Complete draft cover letter | must |
| In-UI editing | Generated draft | Editable text area for refinement | must |
| Tone matching | Same tone preferences as resume | Consistent voice across documents | should |
| Multiple format export | User selects format | PDF, DOCX, plain text, Markdown | must |
| Version history | Each cover letter iteration | Saved drafts with timestamps | could |

### 4. Job Input

How job postings enter the system.

| Capability | Input | Output | Priority |
|-----------|-------|--------|----------|
| Paste job description | Raw text or URL | Structured job data (title, company, requirements, nice-to-haves) | must |
| Job parsing with Claude | Unstructured job posting text | Extracted: role, company, required skills, preferred skills, salary range, location, remote status | must |
| Job match scoring | Parsed job vs knowledge bank | Match score with breakdown (skills match, experience match, gaps) | should |
| Save job for later | Any parsed job | Stored in local DB for future reference | must |

**Note:** API integrations with job boards (LinkedIn, Indeed, aggregators) are deferred post-MVP. MVP is manual paste-in only.

### 5. Application Tracking

Basic status tracking with outcome data for the learning loop.

| Capability | Input | Output | Priority |
|-----------|-------|--------|----------|
| Status tracking | User updates status | Applied / Interview / Offer / Rejected per application | must |
| Link documents | Application entry | Associated resume version + cover letter version used | must |
| Outcome recording | Status changes | Timestamped status history per application | should |
| Application list view | All applications | Filterable/sortable list with status, company, role, date | must |

### 6. Learning Loop

The agent improves over time based on feedback and outcomes.

| Capability | Input | Output | Priority |
|-----------|-------|--------|----------|
| User feedback on generated docs | Thumbs up/down or rating on resume/cover letter | Stored preference signal | should |
| Outcome correlation | Which resume/cover letter versions led to interviews vs rejections | Weighted patterns (skills emphasis, tone, length that worked) | should |
| Style refinement | Accumulated feedback | Future generations incorporate learned preferences | could |

**Note:** The learning loop can be simple for MVP — store the feedback data and correlations. Sophisticated ML-style learning is post-MVP.

### 7. Offline Mode

Opt-in offline agent setup for users without LLM API accounts.

| Capability | Input | Output | Priority |
|-----------|-------|--------|----------|
| Offline mode toggle | User selects "Use Offline Agents" in settings | Downloads required models (Sentence Transformers, spaCy) | must |
| System requirements check | User's machine specs | Warning: "Offline mode requires ~500MB disk, ~2GB RAM. Continue?" | must |
| Automated setup | User confirms | Downloads and configures models silently, shows progress | must |
| Graceful fallback | Any feature request when offline | Uses algorithmic path (no LLM), UI indicates "offline mode" | must |
| Upgrade path | User later adds an LLM API key | Seamlessly switches to LLM-enhanced mode, keeps all data | should |

### 8. Match Calibration (Judgement System)

Users rate match quality to calibrate the scoring system — similar to how judgement games collect preference data.

| Capability | Input | Output | Priority |
|-----------|-------|--------|----------|
| Match report generation | Job + knowledge bank match | Report showing % match with breakdown (skills: 80%, experience: 60%, gaps: [...]) | must |
| User judgement on match | User rates: "Was this a good match?" (yes/partially/no) | Stored calibration signal | must |
| Calibration data storage | Accumulated judgements | Local calibration dataset (job features → match quality → user rating) | must |
| Score adjustment | Calibration dataset | Matching weights adjusted per user (e.g., user values skills match more than years of experience) | should |
| Calibration export | User request | Anonymized calibration report (no PII — only feature weights and match patterns) | should |
| Bulk calibration | User reviews multiple matches | Swipe-style or quick-rate interface for efficient judging | could |

**Privacy:** Calibration data is stored locally. Export strips all PII (company names, job titles, personal details) — only match feature vectors and ratings are included. No data leaves the device unless user explicitly exports.

## Dependencies

- LLM Provider (any of: Anthropic, OpenAI, Ollama, HuggingFace) — for AI-powered extraction, generation, matching. Optional — app works without it.
- Sentence Transformers (all-MiniLM-L6-v2) — semantic sentence/paragraph similarity (~80MB model). Core of offline matching.
- scikit-learn — TF-IDF, cosine similarity. Lightweight, no model needed.
- spaCy (en_core_web_md) — entity extraction (companies, titles, skills) from text (~50MB model).
- RapidFuzz — fuzzy string matching ("React.js" ≈ "ReactJS").
- Tauri 2.0 — desktop app shell
- Python/FastAPI — backend API
- React/TypeScript — frontend UI
- SQLite — local data storage
- PDF generation library (e.g., WeasyPrint, reportlab) — for resume/cover letter PDF export
- DOCX generation library (e.g., python-docx) — for Word format export
- PDF/DOCX parsing library (e.g., PyMuPDF, python-docx) — for resume import
