# Requirements: User Authentication & Multi-User Support

> Generated by /requirements on 2026-04-29
> Mode: standard
> Scope: infrastructure (auth + data isolation for existing app)
> User level: 3 (developer)
> Project state: existing (adding auth to single-user app)

## Problem Statement

The app is currently single-user with one SQLite DB, no auth, and API keys stored in plaintext. To host it for others (or even share with friends), it needs user accounts, data isolation, and secure key storage.

## Core Requirement

Any user can sign up, log in, and use the full app with their own isolated data — without seeing or affecting other users' data.

## Boundaries
- **Excluded:** OAuth/social login (future), payments, admin dashboard, user-to-user interaction
- **Constraints:** Must not break existing single-user functionality. Existing data should be migratable to the first user account.

## User Stories

- As a new user, I want to sign up with email + password so I can start using the app
- As a returning user, I want to log in and see my data from last session
- As a user, I want my resumes, KB, jobs, and settings completely separate from other users
- As a user, I want to enter my API keys once and have them stored securely
- As a user, I want to update my profile (name, email) in settings

## Functional Requirements

### 1. Authentication

| Capability | Input | Output | Priority |
|-----------|-------|--------|----------|
| Sign up | Email + password | Account created, logged in, JWT token | must |
| Log in | Email + password | JWT token, redirect to app | must |
| Log out | Button click | Token cleared, redirect to login | must |
| Password hash | Raw password | bcrypt hash stored in DB | must |
| JWT token | Login success | Token with user_id, expiry (24h) | must |
| Protected routes | Request without valid JWT | 401 Unauthorized | must |
| Token refresh | Approaching expiry | New token issued | should |
| Forgot password | Email | Reset link (future — park for now) | could |

**Auth flow:**
```
Sign up: email + password → validate → hash password → create user → create user DB → issue JWT
Login: email + password → verify hash → issue JWT
Every request: JWT in Authorization header → validate → extract user_id → route to user's DB
```

### 2. User Profile

| Capability | Input | Output | Priority |
|-----------|-------|--------|----------|
| View profile | GET /api/auth/me | name, email, created_at | must |
| Update name | PUT /api/auth/me | Updated profile | must |
| Change password | Old + new password | Password updated | should |
| Delete account | Confirmation | User + all data deleted | should |

### 3. Per-User Data Isolation

| Capability | Input | Output | Priority |
|-----------|-------|--------|----------|
| Separate SQLite DB per user | User signup | `~/.sahaiy/users/{user_id}/data.db` | must |
| User DB created on signup | Registration | Empty DB with full schema (migrations run) | must |
| Request routing | JWT user_id | Connect to correct user DB | must |
| Migrate existing data | First admin user | Current `sahaiy.db` becomes user 1's DB | should |

**DB structure:**
```
~/.sahaiy/
  auth.db              ← shared: users table (email, password_hash, created_at)
  users/
    {user_id}/
      data.db           ← per-user: all existing tables (jobs, resumes, KB, settings, etc.)
```

### 4. Secure API Key Storage

| Capability | Input | Output | Priority |
|-----------|-------|--------|----------|
| Encrypt API keys before storage | Plaintext key | AES-256 encrypted in user's settings table | must |
| Decrypt on use | Encrypted key from DB | Plaintext for API call, never logged | must |
| Server encryption key | `ENCRYPTION_KEY` env var | Used for AES encrypt/decrypt | must |
| Key never in logs | Any logging | Keys masked as `sk-ant-***` | must |

**Encryption:**
- Algorithm: AES-256-GCM (authenticated encryption)
- Key: 32-byte key from `ENCRYPTION_KEY` env var (or auto-generated on first run)
- Each user's API keys encrypted with the same server key
- Decrypted only in memory when making LLM calls

### 5. Frontend Auth Flow

| Capability | Input | Output | Priority |
|-----------|-------|--------|----------|
| Login page | Email + password form | JWT stored in memory/cookie | must |
| Signup page | Email + password + name form | Account created, redirected to app | must |
| Auth guard on all routes | No valid token | Redirect to /login | must |
| Logout button | Click | Token cleared, redirect to /login | must |
| Profile section in Settings | Name, email display | Editable fields | must |

## Non-Functional Requirements

| Category | Requirement | Target |
|----------|------------|--------|
| Security | Password storage | bcrypt, cost factor 12 |
| Security | JWT expiry | 24 hours |
| Security | API key encryption | AES-256-GCM |
| Security | HTTPS in production | Required for hosted deployment |
| Performance | Login response | < 500ms |
| Data | User data isolation | Complete — no cross-user queries possible |
| Data | Account deletion | Hard delete — user DB file removed |

## Data Requirements

| Data | Source | Stored? | Sensitive? | Location |
|------|--------|---------|-----------|----------|
| Email | User input | Yes | Yes (PII) | auth.db |
| Password hash | bcrypt(password) | Yes | Yes | auth.db |
| User name | User input | Yes | Yes (PII) | auth.db |
| JWT token | Server-generated | No (stateless) | Yes | Client only |
| LLM API keys | User input | Yes (encrypted) | Yes | user's data.db |
| Encryption key | Server env var | No (env only) | Critical | `.env` |

## API Endpoints (New)

```
POST   /api/auth/signup     { email, password, name }  → { token, user }
POST   /api/auth/login      { email, password }        → { token, user }
GET    /api/auth/me                                     → { id, email, name }
PUT    /api/auth/me          { name }                   → { updated user }
POST   /api/auth/logout                                 → { ok }
DELETE /api/auth/me          { password }               → { deleted }
```

All existing `/api/*` endpoints remain unchanged but require valid JWT.

## Assumptions

- 10-50 users max — SQLite per user is appropriate (no PostgreSQL needed)
- No email verification on signup (can add later)
- No forgot-password flow (can add later)
- No rate limiting on auth endpoints (can add later for abuse prevention)
- Server runs behind HTTPS in production (nginx/caddy)
- Single server deployment (no horizontal scaling needed)

## Dependencies

- `bcrypt` — password hashing (already available via `passlib`)
- `pyjwt` — JWT token creation/validation
- `cryptography` — AES-256-GCM for API key encryption
- All are well-known, actively maintained packages

## Reusable from Codebase

- `shared/db.py` — `connect_sync()` already accepts `db_path` parameter, works with per-user paths
- `shared/db.py` — `migrate()` runs all migrations on any new DB
- `backend/main.py` — already has a `_conn` global that can be replaced with per-request connection
- Profile table already exists with user fields

## Implementation Order

1. **Auth DB + user model** — `auth.db` with users table
2. **Signup/login endpoints** — bcrypt + JWT
3. **Auth middleware** — validate JWT on every request, resolve user DB path
4. **Per-user DB routing** — `connect_sync(user_db_path)` per request
5. **API key encryption** — encrypt on save, decrypt on use
6. **Frontend auth pages** — login, signup, auth guard
7. **Profile UI** — name/email in Settings
8. **Migration** — move existing single-user data to first account
